guarantee · reversible

Reversible,
gated, versioned state

High-impact actions are gated, versioned, and recoverable. Nothing committed is ever edited in place: apply a change, try to force one through ungated, try to reach past the API, then recover a prior state below.

the claim

Recovery is deterministic, and history cannot be rewritten

apply(change) is the only sanctioned mutation: it appends a NEW version, current state merged with the change (supersede-forward). Predecessors are never touched. An ungated high_impact change is refused outright. restore(version=N) recovers a prior state the same way, by appending a new version equal to it, so its state_hash matches version N's exactly, byte-for-byte. There is no update() or delete() on a committed version, reaching past the API fails closed.

the version chainSHA-256, computed in your browser

Apply commits a new version forward. The ungated and in-place buttons each attempt to reach past that sanctioned path, and each is refused for a different reason: a missing approval token, or the durable append-only guard. Restore reconstructs a prior state and its hash is compared against the original snapshot.

Don't take this at face value

This page is a JavaScript reimplementation of demos/reversible-actions/reversible.py. It uses the same versioning rule, so the hashes match the Python byte-for-byte, but a demo you cannot inspect proves nothing. The real proof is the code you run yourself:

git clone https://github.com/frapster/the-substrate && cd the-substrate
python demos/reversible-actions/reversible_demo.py     # watch it gate, guard, and recover
python demos/reversible-actions/test_reversible.py     # the negative tests are the point

Prove it yourself · extend the tests

The tests in demos/reversible-actions/test_reversible.py assert that recovery is deterministic and that every attempt to rewrite history is refused for the right reason. Add your own case and confirm it holds. Drop this into the GuardsEnforced class and re-run:

def test_my_own_restore(self):
    store = _seed()
    store.apply({"tier": "silver"}, gated=True)   # apply N versions
    store.apply({"tier": "bronze"}, gated=True)
    target = 2                                     # restore an arbitrary earlier one
    checkpoint_hash = store.versions[target].state_hash
    restored = store.restore(target)
    self.assertEqual(restored.state_hash, checkpoint_hash)   # must match exactly

Run python demos/reversible-actions/test_reversible.py. If you can make an in-place edit that the append-only guard misses, or a restore that does not reproduce the exact prior hash, you have broken the claim. That is the point: it is falsifiable, and you hold the falsifier.
