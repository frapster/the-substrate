spine · AI : code ratio

Where does new
requirement meaning land?

One feature, support-ticket routing by plan tier and severity, built two ways below: a hand-authored if/elif tree, and a generic evaluator reading a fact table. Add a new business case and watch what it costs each one.

the claim

New cases cost the code path branches, and the governed path facts

Both implementations agree on the base requirement (3 plan tiers × 3 severities, 9 tickets). When three new business cases arrive (critical severity), the code version needs new elif branches, a developer writes, reviews, and ships them. The governed version needs new rows in a fact table; its one evaluator function never changes. Add a case below and watch the counters move.

the routing tablecode vs. governed, live in your browser

"Add a new case" adds one critical-severity case to both sides: the code version gets a new elif branch (counted), the governed version gets a new fact row. "Route an ungoverned ticket" sends a ticket neither side was ever built to handle (severity=outage): the governed evaluator abstains; the unmodified code falls through to its default, silently.

Don't take this at face value

This page is a JavaScript reimplementation of demos/ai-code-ratio/code_version.py, code_version_v2.py, facts.py, and governed_version.py. The branch and LOC counts shown here are hardcoded to match what the Python actually measures by parsing its own source with ast, they are not recomputed live from this JS. A demo you cannot inspect proves nothing; the real proof is the code you run yourself:

git clone https://github.com/frapster/the-substrate && cd the-substrate
python demos/ai-code-ratio/ratio_demo.py     # watch the counted delta: 9 → 12 branches
python demos/ai-code-ratio/test_ratio.py     # the revert-the-fact litmus is the point

Read the source: code_version.py · governed_version.py · README · THESIS.md

Be honest about scope: this is one small, illustrative feature, not a whole-system measurement. The 90:10 LLM-First target is a design aspiration, not an empirical finding about this or any system. See the README's "what this does and does not prove" section.

Prove it yourself · extend the tests

Adding a business case to the governed path means adding a row to NEW_FACTS in facts.py, the evaluator in governed_version.py is untouched. Add your own case and assert it changes behavior with zero evaluator code change:

NEW_FACTS: list[dict] = [
    {"conditions": {"plan_tier": "free", "severity": "critical"},
     "outcome": {"queue": "standard_urgent", "escalation": "tier1"}},
    # ...existing rows...
    {"conditions": {"plan_tier": "pro", "severity": "outage"},   # your new case
     "outcome": {"queue": "priority_urgent", "escalation": "tier4_incident"}},
]

def test_my_own_fact_changes_behavior(self):
    probe = {"plan_tier": "pro", "severity": "outage"}
    before = governed_version.evaluate(probe, facts=facts.BASE_FACTS)
    after = governed_version.evaluate(probe, facts=facts.BASE_FACTS + facts.NEW_FACTS)
    self.assertEqual(before.get("status"), "hitl_required")   # abstained before
    self.assertNotEqual(after.get("status"), "hitl_required") # governed after

Run python demos/ai-code-ratio/test_ratio.py. If your new fact row changes the routing decision without touching governed_version.py, you've reproduced the claim. Then try reverting just that one row, the revert-the-fact litmus, and confirm the decision reverts too.
