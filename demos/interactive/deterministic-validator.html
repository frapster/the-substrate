Deterministic
validator commits

A model proposes an action. Deterministic code grades it against real, checkable rules. Only a PASS commits. A rejection is discarded, not patched, and an approving judge cannot force it through. Try to launder a hallucination below.

the claim

The model is never trusted, it is checked

validate() is plain, hand-written code: is the cited source in the known set, is the amount within policy, are required fields present. No model call sits inside the check. A proposal that fails is discarded exactly as submitted, never rewritten into one that passes, and an llm_judge() can feed evidence into the review but has no path to the commit log, only validate() does. Build a proposal, optionally let the judge weigh in, then submit.

The judge toggle runs a mocked second model that only eyeballs the amount and never checks the source, it will happily approve a hallucinated citation. Its approval is logged for context but submit() never reads it: only validate() decides what commits.

Don't take this at face value

This page is a JavaScript reimplementation of demos/deterministic-validator/validator.py. It uses the same checks and the same reasons, but a demo you cannot inspect proves nothing. The real proof is the code you run yourself:

git clone https://github.com/frapster/the-substrate && cd the-substrate
python demos/deterministic-validator/validator_demo.py   # watch a hallucinated proposal get discarded
python demos/deterministic-validator/test_validator.py   # the negative tests are the point

Prove it yourself · extend the tests

The tests in demos/deterministic-validator/test_validator.py assert that validate() fails for the right reason, and that an approving judge cannot force a bad proposal through. Add your own attack and confirm the gate catches it. Drop this into the RejectionIsClean class and re-run:

def test_my_own_attack_is_still_discarded(self):
    validator = Validator()
    bad = propose(intent="grant_refund", source_id="src_999", amount_usd=40)
    judgment = llm_judge(bad)              # the judge is fooled
    self.assertTrue(judgment.approved)
    result = validator.submit(bad, judgment)
    self.assertFalse(result.ok)            # the deterministic gate is not fooled
    self.assertNotIn(bad, validator.commit_log)
    committed_ids = [p.source_id for p in validator.commit_log]
    self.assertNotIn("src_999", committed_ids)   # never laundered in

Run python demos/deterministic-validator/test_validator.py. If you can find a way to make an approving judge commit a bad proposal, or a way for a rejected proposal to land in commit_log in any form other than byte-identical to what was proposed, you have broken the claim. That is the point: it is falsifiable, and you hold the falsifier.

This interactive page is a clean-room illustration, not the BOSNet.io engine. It demonstrates one gate primitive (deterministic commit admissibility) with code you can read and rerun.
