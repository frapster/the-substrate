01 dimensional governance strategy

Reasoning compiled from a governed model — not hand-wired

The strategy, stated at the level appropriate for a public document: operational behavior and surfaces are compiled from a governed dimensional model, every output evidence-backed and audited, with human-in-the-loop approval on high-impact actions. Content and decisions change only through governed events — never ad-hoc edits — so a change to one modeled facet propagates surgically to everything that references it.

Patent pending

What is public: the shape and its guarantees. Run-time reasoning sits inside a governance envelope that is bounded, evidence-backed, audited, and reversible. The audit backbone is an append-only, hash-chained ledger with recomputable verification and update/delete-blocking — facts are versioned, never silently overwritten. Accessibility and cost are treated as governance constraints, not afterthoughts.

What is reserved: the governance kernel and compiler, the dimensional model and its coordinate scheme, and the machine-checkable specification. Those are patent-pending and described only conceptually here. See BOSS-STANDARD.md and THESIS.md.

The deeper claim: an LLM's operational substrate is governance + knowledge — not code. Governance bounds what the model may do; the dimensional knowledge model determines what it correctly knows. Code is what you write when you lack the other two.

02 knowledge bases

The KB structure follows the retrieval job

Every project's knowledge base is shaped by what it will be asked. None is a flat pile of chunks.

Today Series

Four-facet KB + relationship graph

The same source is stored at four granularities — prose, atomic facts, Q&A pairs, taxonomy — plus a typed cross-reference graph (duplicate · nuance_variant · contradicts · extends) with an LLM-written note. Stores knowledge at multiple resolutions and the relationships between claims.

job-hunt engine

Layered, provenance-linked

Physical document chunks → hashed evidence → promoted atomic facts → a typed knowledge graph of skills/orgs/roles. A "dimensional twin" of a career, every fact evidence-backed and coordinate-addressed. (concept only)

Eikon Digital

Per-query-rendered knowledge

KB entries are not standalone pages — they're a knowledge dimension rendered into responses per query, so knowledge and presentation stay separate and governed.

Zabble

Entities on an event spine

No separate KB: entities are the addressing scheme on an append-only event stream, and read models are rebuilt (CQRS-lite) from that stream rather than maintained by hand.

03 vector embeddings

Chunks vs. atomic facts vs. Q&A — a deliberate "don't just chunk" stance

The most-copied RAG recipe is "split into chunks, embed, retrieve." The choice here is to embed at the granularity that matches the question's shape — and, where the fact set is closed, to use no vectors at all.

prose (chunked)

~800-token paragraphs, 15% overlap, only 2–4 per article. Preserves context for open-ended queries — without over-fragmenting.

atomic facts

One claim per row. Answers precise point questions cleanly, where a big chunk would bury the answer in noise.

Q&A pairs

Matches conversational queries directly — the retrieval shape closest to how a chatbot is actually asked.

taxonomy

Topics/themes/keywords — powers near-duplicate detection and topical routing, not just answer retrieval.

the decision, in one line

Granularity is a retrieval decision, not a default. Atomic facts for point questions, Q&A for conversation, prose for context, taxonomy for routing — and no embeddings at all when the fact set is closed and determinism is the product.

and for governed knowledge

Naive vector similarity returns what is near, not what is governed and true — retrieval doesn't remove hallucination (RAGTruth), degrades mid-context even on a correct hit (Lost in the Middle), and turns every retrieved document into an injection surface (OWASP LLM08:2025). So the rule is right index per query class: hybrid RAG is fine for open factoid lookup, but anything a decision must be accountable for wants governed, structured knowledge — the direction GraphRAG validates. 04 model optimization

Cheapest viable tier by default — escalate on deterministic signals

One swappable task→model catalog per system; the cheapest model that can do the job runs first; escalation is triggered by facts the code can verify, never by the model's own confidence.

Relic Wars

nano → mini cascade, empirically derived

A 200-contest bakeoff across four models with real cost measurement. gpt-5-nano primary (~$0.20/1k contests), auto-escalate to gpt-5-mini on referee rejection. Blended governed cost ≈ $0.25/1k — about half of running mini for everything.

Today Series

small-for-extraction, large-for-authoring

gpt-4o-mini at temp 0.2 for fact/Q&A/taxonomy extraction and the nuance judge; Claude for authoring and the chatbot; OpenAI for embeddings and moderation. Concurrency tuned per stage.

Zabble

gateway router + spend governance

A TASK_MODEL catalog of provider/model strings behind an AI gateway — one-line re-tuning, no call-site edits. Per-group budgets, a spend ledger, and hard caps gate every AI action.

the sharp insight

In the bakeoff, the cheap model invented an entity ID ~10% of the time — confidently, and unflagged. So the escalation gate is deterministic (schema validity, entity-ID membership, citation legality), never the model's needs_review self-report. And the metric is cost per successful op, not sticker price: nano burns 4–5× more reasoning tokens than mini and only wins because its per-token price is ~8× lower — "a smaller price gap could flip the ordering."

05 wordpress lineage

Data-driven since the plugin era — and the deliberate exit from it

The schema-driven, query-driven instinct visible in the Supabase work started in WordPress. The move to Next.js/Supabase was a reasoned architectural rejection, not a trend-chase.

The stack

Data-driven WordPress, not page-builder soup

A JetEngine/Crocoblock architecture — custom post types, relationships, query builders, and faceted filtering (JetSmartFilters) — on a lightweight Elementor base with a custom Tailwind child theme. Pairing WordPress with Tailwind well before the React era.

Custom plugins — DAM, CRM capture, API manager

Not just building pages: custom WordPress plugins for digital asset management, CRM contact capture, a global API manager, access control, and scanning — infrastructure work inside the CMS.

The exit

The "anti-WordPress" move — compile once, serve at the edge

The migration to Next.js/Supabase is framed explicitly as rejecting the CMS "query the DB and render a template on every request" model in favor of compile-once → static/ISR at the edge, dynamic only where structurally necessary (chat, auth, forms). A clean four-tier compilation model, with WCAG accessibility treated as a governance constraint.

Next.js · RSCSupabasestatic / ISRedge
The bridge

Integrate, don't rip-and-replace

Rather than abandon WordPress outright, a bridge re-casts the WP admin as a thin render-tree client talking to a Next.js/Supabase backend through a same-origin REST proxy — with per-user auth provisioning and an HMAC token handoff (no third-party cookies).

06 maturity signals

Patterns that show up when the stakes are real

The choices that separate a demo from a system you can operate.

Event sourcing in Postgres

An append-only domain_events spine with a single write path, trigger fan-out to queues, and CQRS-lite read models — "the Kafka replacement, inside Postgres." Enforced against UPDATE/DELETE and TRUNCATE.

Zabble

Server-authoritative, seeded resolve

A seeded PRNG (never Math.random() in engine code), a pure resolve function server-side, outcomes emitted as a draw-only script. Reproducible and fair.

Relic Wars

RLS everywhere, deny-by-default

Row-level security on every table, engine access via service-role only, app reads through wrapper views. Security is the default state, not a step.

Today Series · job-hunt · Zabble

Security-definer RPCs as the authz surface

Gated SECURITY DEFINER functions with pinned search_path instead of relaxing RLS — refunds, hybrid search, privileged reads all go through a controlled door.

Zabble · Today Series

Governance that can't be forgotten

A DDL trigger auto-enables RLS on every new table; new users auto-provision profile rows. The safe thing happens by default.

Relic Wars

Edge-only credential isolation

Tenant/processor secrets resolved only inside edge functions (Supabase Vault), never in a route handler; presence-only status functions that never return the value.

Zabble · job-hunt

Graceful degradation as posture

Every integration returns a clean 503/no-op when its key is absent — "nothing crashes; everything lights up when keys are added."

Today Series

Append-only, hash-chained audit

A SHA-256 hash-chained ledger with recomputable verification and update/delete-blocking; facts versioned via supersede, never overwritten.
