the conversion: before → after

Adjudication subsystem

Before

100% code

A ~2,350-line rules engine handled every adjudication decision.

After

~85% governed reasoning · ~15% code

The engine is retired. A fact matrix and prompt drive the model; ~15% of the surface stays deterministic code.

~88% of adjudication code displaced. factual · shipped
Code computes what does happen; the substrate decides what should happen; a deterministic auditor grades every decision. Math is law; code disposes.
the governance loop

Every contest passes a commit gate

~8 decisions per snap route through the model in dependency order. None commit unless a deterministic validator confirms the model cited only real, in-envelope facts and did not contradict the computed outcome.

Fail-closed. The auditor never "corrects" the model. A bad answer is thrown away. Twice-failed contests commit nothing.

total cost of ownership

The part that changes most is now the cheapest to change

A rules engine's lifetime cost is dominated by enhancement (endless balancing and new content), the expensive 60% of the 60/60 maintenance rule. The conversion attacks exactly that bucket.

~88%↓

Adjudication build cost: ~2,350 → ~280 lines.

1 row

Adding a new rule: one table row instead of a new code branch.

60/60

Maintenance ≈ 60% of lifetime cost; enhancement ≈ 60% of that: the slice this targets.

code quality & security

Never trust the client or the model

The category norm and the AI-governance model are the same shape: authoritative state stays server-side, and every answer is re-derived and checked.

Category standard

server-authoritative never trust the client never trust the model

LLM-authored code is a draft until graded: see ENGINEERING.md §3.

Governance in place

  • Deterministic auditor: fail-closed, two-tier escalation, denial receipts
  • Server-authoritative resolve: runs code-only with no model key
  • Auto-RLS DDL trigger: RLS enabled on every new table
  • 30 test files + a 900-match balance tournament harness

Where code stayed

The numeric core

All arithmetic, seeded randomness, and geometry: the numeric core. Its bulk is algorithmic, so it is the least blueprint-compilable of the four, the honest anchor for the estimates elsewhere.
