the substrate move

A 2,350-line engine became a table, a prompt, and a validator

The original match logic lived in one monolithic play engine — branchy rules dispatch. It was retired. The adjudication "brain" is now three small parts that a model drives at run-time.

Retired

Monolithic play engine

~2,350 LOC
  • Enumerated rule branches
  • Every capability = new code path
  • Outcome logic entangled with dispatch
  • Hard to explain after the fact
replaced by
Governed substrate

Fact table + prompt + auditor

~280 LOC
  • ~120-line capability fact table
  • ~70-line adjudication prompt
  • ~90-line deterministic auditor
  • New rule = one table row
Code computes what does happen. The substrate decides what should happen. The contest is the seam — and a deterministic auditor grades every decision against seeded ground truth. Math is law; code disposes.
the governance loop

Every contest passes through a commit gate

Per snap, ~8 decisions route through the model in dependency order. None commit unless a deterministic validator confirms the model cited only real, in-envelope facts and did not contradict the computed outcome.

Fail-closed. The auditor never "corrects" the model — a bad answer is thrown away. Twice-failed contests commit nothing and record a denial receipt.

how the substrate is built

Six load-bearing pieces

01 · DIMENSIONAL MODEL

Coordinate-addressed matrix

Contest facts compiled to path-addressed nodes — entity/{id}/aptitude, read/{id}/roll — pinned by a deterministic compile hash. The model only ever sees this closed, hashed table.

02 · KNOWLEDGE GRAPH

Capabilities as rows

Each capability maps an aptitude stat → opposed stat → consequence. Plus a self-pruning read/counter-read graph, line-of-sight geometry, and a per-snap dependency DAG.

03 · VECTORIZED DATA

None — by design

No embeddings, no retrieval. Reasoning runs over a closed fact set, not an open corpus. Governance-by-envelope is what lets the auditor verify every claim.

04 · GOVERNANCE

The Auditor

Rejects any output that invents an entity, cites out-of-envelope facts, omits citations, or contradicts the computed outcome. Fail-closed; never patches.

05 · EVIDENCE

Append-only ledger

Every contest writes compile hash, seed, model ref, causal chain, and verdict. Server-write-only; a player reads their own match but cannot write the ledger.

06 · RESOLVE

Server-authoritative

Resolves once, server-side, as a resumable job, idempotent per play. Governance is additive — with no model key the match still resolves fully in code.

the honest boundary

Where it paid off

Rule extensibility (rows, not branches) and explainability — every outcome ships with checkable citations against a hashed fact set.

Where code stayed, deliberately

All arithmetic, seeded randomness, and geometry. The numeric core is exactly where certainty matters — so it stayed in code.
