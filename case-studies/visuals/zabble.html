Zabble's product is its deterministic envelope. A few reasoning calls can be trusted because the code around them is rigid — so the conversion stops well short of the 90 : 10 target, on purpose.
the conversion roadmap

Govern the judgment seams — freeze the core

Apply the envelope/compiler/auditor pattern only to the reversible, judgment-heavy surfaces. This is a plan; none of it has shipped.

1

Blueprint read models & discovery

Replace hand-tuned ranking and read-model assembly with blueprint-compiled reasoning over the existing embeddings and graph.

2

Widen the governed moderation seam

Move more borderline moderation and pattern analysis into governed reasoning — fixed action space, human escalation — but keep L1 deterministic.

3

Compile the extraction fleet

Blueprint-driven, auditor-checked ingestion — one blueprint per source shape, not per site.

4

Freeze the core

Payments, the L1 safety gate (cannot be prompt-injected), and the event-sourced spine stay deterministic by policy. This is the ceiling — and it is intentional.

code quality & security

The strictest external bar of the four

Payments, trust-and-safety, and multi-tenant access control — exactly the categories where the published weakness of ungoverned LLM code bites hardest.

Category standard

OWASP Top 10:2025 A01 broken access control (#1) OWASP LLM Top 10 PCI DSS 4.0

Ungoverned LLM code picks the insecure option ~45% of the time — ENGINEERING.md §3.

Governance in place

  • Security findings F-01…F-12 closed across tracked batches + a plan-drift self-heal
  • CI service-role-key guard — fails the build if a privileged secret escapes its module
  • pgTAP + RLS-audit skill on an ephemeral database
  • Three-layer safety: deterministic L1 gate, fenced L2 model, batch L3
the honest boundary

Where it should expand

Reversible judgment — ranking, curation, moderation triage, extraction.

Where it must not

Anything fast, certain, or irreversible. The most credible thing Zabble can say is where it chooses not to apply the substrate.
